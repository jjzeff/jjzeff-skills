## 项目词表自检流程

> **每次翻译任务开始时，必须先执行此流程。**

### 第一步：识别项目名称

从用户输入中提取游戏/项目名称，例如："穿越火线"、"CrossFire"、"PUBG"、"王者荣耀"、"CF"、"LOL"等。
若用户只说"我们的游戏"等模糊描述，跳过词表检测，直接翻译。

### 第二步：检测词表文件是否存在

检查路径：`references/projects/{项目英文名小写}-glossary.md`

**情况 A：词表存在** → 加载词表，继续翻译，专有名词优先使用词表中的对照。

**情况 B：词表不存在** → 停止翻译，向用户发送以下引导：

```
📋 我没有找到「{项目名称}」的专属词表。

为了确保武器名、角色名、活动名等专有名词翻译准确一致，建议你提供以下信息：

必填（至少提供一项）：
- 核心专有名词对照（武器 / 角色 / 地图 / 模式名称）
- 活动或 UI 文案中的关键术语

格式建议（Markdown 表格）：

| 中文 | 目标语言 |
|------|---------|
| 幽灵模式 | （请填写） |
| 精英特训 | （请填写） |

你也可以直接粘贴已有的对照表，或告诉我「暂时没有，先用通用翻译」。
提供后我会帮你保存成词表文件，下次直接加载。
```

**情况 C：用户回复「暂时没有，先用通用翻译」** → 正常翻译，结果末尾加注：
> ⚠️ 本次使用通用词汇翻译，未加载项目词表。如需锁定专有名词，随时提供对照表。

**情况 D：用户提供了词表内容** →
1. 将内容整理为标准 Markdown 表格
2. 写入 `references/projects/{项目名}-glossary.md`
3. 回复：`✅ 词表已保存，本次翻译将优先使用以上术语。`
4. 加载词表，正常翻译

## 核心原则

- **语气匹配优先**：不同场景语气差异巨大——商务用합쇼체，游戏UI用명사형/해체，活动文案用해요체，社区用반말；默认先判断场景再翻译
- **自然表达 > 字面直译**：中文直译韩文往往生硬，优先找韩国本地对应表达方式
- **保留行业惯用词**：英文缩写（DAU/ROI/IP/SDK）和外来语游戏术语（아이템/스킨/가챠）直接保留
- **双语对照输出**：翻译结果提供原文+译文对照，遇到多种译法时给出推荐并说明理由

## 场景识别与工作流

### 快速场景判断
1. **游戏文案/本地化**（UI、活动、剧情、推送）→ 先读 `casual-localization.md`
2. **商务沟通**（邮件、提案、合同）→ 先读 `business-communication.md` + `game-terms.md`
3. **专业术语**（运营指标、研发词汇）→ 读 `game-terms.md`
4. **混合场景**（如"写一封介绍游戏的提案邮件"）→ 同时读相关文件

---

### 工作流 A：游戏文案本地化（通俗韩文）

1. 判断文案类型：UI按钮 / 系统提示 / 活动公告 / 剧情对话 / SNS推文
2. 读 `casual-localization.md`，确认适用语气和本地惯用表达
3. 翻译时重点检查：
   - 是否有直译陷阱（参考文件第6节）
   - 语气词和标点是否符合韩文习惯
   - NPC对话是否匹配角色设定（活泼/端庄/反派等）
4. 输出：原文 + 译文，附简短说明（如选词理由、语气说明）

### 工作流 B：中→韩通用翻译

1. 判断内容类型，选择对应参考文件
2. 翻译时注意：
   - 数字：韩文用 만(万)/억(亿)
   - 货币：人民币标注 위안(CNY)，韩元标注 원(KRW)
   - 日期：YYYY년 MM월 DD일
3. 输出：原文 + 译文，必要时附注

### 工作流 C：韩→中翻译

1. 判断是否有行业术语或特殊语气需精准处理
2. 不确定的表达给出多种解读并标注置信度
3. 商务文件保持正式书面语

### 工作流 D：审校/润色

1. 检查语气等级一致性（全文统一）
2. 检查是否有明显直译生硬之处（参考 `casual-localization.md` 第6节）
3. 检查行业术语是否用韩国本地通用表达
4. 给出修改建议，简要说明原因
